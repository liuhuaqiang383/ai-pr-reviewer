# AI PR Review 助手 - 设计文档

## 1. 系统概述

AI PR Review 助手是一个基于Claude AI的智能代码评审系统，旨在帮助开发者更高效地进行Pull Request审查。系统通过自动化分析代码变更，识别潜在问题，提供改进建议，从而提升代码质量和审查效率。

## 3. 核心模块设计

### 3.1 GitHub服务模块

**职责**: 与GitHub API交互，获取PR相关数据

**关键功能**:
- PR URL解析：支持多种URL格式
- PR元数据获取：标题、描述、作者、分支等
- 文件变更列表：获取修改的文件及统计
- Diff内容获取：完整的代码变更
- 文件内容获取：用于深度分析

**设计考虑**:
- 分页处理：支持大型PR的文件列表
- 错误处理：网络异常、API限流等
- Token管理：支持认证和匿名访问
- 数据缓存：减少重复API调用（未来优化）

```python
class GitHubService:
    def parse_pr_url(self, pr_url: str) -> Optional[Tuple[str, str, int]]:
        """解析PR URL，提取owner、repo、PR号"""

    def get_pr_info(self, owner: str, repo: str, pr_number: int) -> Dict:
        """获取PR基本信息"""

    def get_pr_files(self, owner: str, repo: str, pr_number: int) -> List[Dict]:
        """获取变更文件列表"""

    def get_pr_diff(self, owner: str, repo: str, pr_number: int) -> str:
        """获取完整diff"""
```

### 3.2 AI分析服务模块

**职责**: 调用Claude API进行代码分析

**提示词工程策略**:

1. **角色设定**: 明确AI作为代码审查专家的身份
2. **上下文提供**: 包含PR元数据、文件概览、代码变更
3. **任务定义**: 明确需要分析的维度
4. **输出格式**: 指定JSON格式便于解析
5. **重点关注**: 强调关键审查点

```python
def _build_analysis_prompt(self, pr_info: Dict, diff: str, files: List[Dict]) -> str:
    prompt = f"""你是一个专业的代码审查专家。请分析以下GitHub Pull Request...

    ## PR信息
    - 标题: {title}
    - 作者: {author}
    ...

    请提供以下分析（使用JSON格式返回）：
    1. summary: PR的整体总结
    2. risk_level: 风险等级
    3. issues: 发现的问题列表
    ...
    """
```

**分析维度**:

| 维度 | 检测内容 | 严重程度 |
|------|----------|----------|
| Bug | 逻辑错误、空指针、边界条件 | critical/high |
| Security | SQL注入、XSS、敏感信息泄露 | critical |
| Performance | N+1查询、内存泄漏、算法复杂度 | medium/high |
| Style | 命名规范、代码格式、注释 | info |
| Architecture | 设计模式、耦合度、可扩展性 | medium |
| Documentation | 文档完整性、注释质量 | info |

**结果解析策略**:
- 优先尝试JSON解析
- 失败时提取结构化文本
- 提供默认值保证前端渲染

## 4. 模型选择设计思路

### 4.1 模型对比

| 模型 | 速度 | 质量 | 成本 | 适用场景 |
|------|------|------|------|----------|
| Haiku | ★★★★★ | ★★★ | $ | 快速预览、简单PR |
| Sonnet | ★★★★ | ★★★★ | $$ | 日常审查（推荐） |
| Opus | ★★ | ★★★★★ | $$$$ | 深度分析、关键PR |

### 4.2 选择策略

**默认选择Sonnet的原因**:
1. **平衡性**: 速度和质量的最佳平衡
2. **成本效益**: 适合日常使用
3. **代码理解**: 对代码有较好的理解能力
4. **响应速度**: 用户体验良好

**动态选择建议**:
```python
def select_model(pr_size: int, complexity: str) -> str:
    if pr_size < 100:  # 小型PR
        return 'claude-haiku-4-20250514'
    elif complexity == 'high':  # 复杂PR
        return 'claude-opus-4-20250514'
    else:
        return 'claude-sonnet-4-20250514'
```

### 4.3 优化方向

1. **分层分析**: Haiku初筛 + Opus精析
2. **缓存机制**: 相同PR不重复分析
3. **增量更新**: 只分析新增提交
4. **并行处理**: 多文件同时分析

```
│           代码级别上下文                 │
│  - Diff内容                              │
│  - 变更前后对比                          │
│  - 周围代码（未来）                      │
└─────────────────────────────────────────┘
```

### 5.2 上下文限制处理

**问题**: Claude API有token限制，大型PR可能超出

**解决方案**:

1. **Diff截断**: 限制最大diff大小（100KB）
2. **文件数量限制**: 最多审查50个文件
3. **智能摘要**: 对大型文件提取关键变更
4. **分批处理**: 超限时分批分析

```python
# 配置限制
MAX_DIFF_SIZE = 100000  # 100KB
MAX_FILES_TO_REVIEW = 50

# 截断处理
if len(diff) > Config.MAX_DIFF_SIZE:
    diff = diff[:Config.MAX_DIFF_SIZE] + '\n... (diff truncated)'
```

### 5.3 上下文增强策略

**当前实现**:
- PR元数据
- 文件列表概览
- 完整Diff

**未来增强**:
1. **相关文件**: 获取import的文件内容
2. **历史变更**: 文件的修改历史
3. **测试覆盖**: 相关测试文件
4. **文档引用**: 相关文档内容
5. **代码规范**: 项目特定的编码规范

## 6. 误报与漏报控制

### 6.1 误报控制

**策略**:
1. **置信度标注**: 每个问题标注置信度
2. **上下文验证**: 结合上下文判断问题真实性
3. **模式匹配**: 使用已知模式减少误判
4. **人工确认**: 提供"这不是问题"的反馈机制

**实现**:
```python
# 问题结构
{
    "severity": "warning",
    "confidence": 0.8,  # 置信度
    "category": "bug",
    "description": "...",
    "suggestion": "...",
    "false_positive": false  # 用户可标记
}
```

### 6.2 漏报控制

**策略**:
1. **多维度检查**: 覆盖不同类型的问题
2. **规则明确**: 在提示词中明确检查点
3. **示例引导**: 提供问题示例
4. **迭代优化**: 根据反馈优化提示词

**检查清单**:

### 6.3 质量保证

**评估指标**:
1. **准确率**: 正确问题 / 总报告问题
2. **召回率**: 发现问题 / 实际问题
3. **F1分数**: 准确率和召回率的调和平均

**优化方法**:

2. **用户反馈**: 收集用户评价
